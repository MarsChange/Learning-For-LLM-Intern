# 大模型资源估算

## 面试定位

应用算法岗不一定要写分布式训练框架，但必须能估算模型训练/推理为什么会 OOM，知道显存花在哪里。

一句话概括：

> 大模型资源主要花在权重、梯度、优化器状态、激活和 KV Cache；训练看前四项，推理重点看权重和 KV Cache。

## 训练显存构成

| 部分 | 是否训练需要 | 说明 |
|---|---|---|
| 参数权重 | 是 | fp16/bf16/fp32 |
| 梯度 | 是 | 与可训练参数同量级 |
| 优化器状态 | 是 | AdamW 通常有一阶/二阶矩 |
| 激活 | 是 | 反向传播需要保存或重算 |
| 临时 buffer | 是 | kernel、通信、loss 等 |

AdamW 全参训练粗略显存：

```text
参数 bf16: 2 bytes
梯度 bf16: 2 bytes
主权重 fp32: 4 bytes
Adam m fp32: 4 bytes
Adam v fp32: 4 bytes
合计约 16 bytes / parameter
```

实际会因框架、ZeRO/FSDP、混合精度策略而变化。

## 推理显存构成

推理主要是：

```text
模型权重 + KV Cache + runtime buffer
```

权重显存：

$$
\text{weight memory}=N \times \text{bytes}
$$

KV Cache：

$$
2 \times L \times B \times T \times H_{kv} \times d_{head} \times \text{bytes}
$$

长上下文和高并发下，KV Cache 往往比想象中更快成为瓶颈。

## FLOPs 粗估

Dense Transformer 训练 FLOPs 常用：

$$
C \approx 6ND
$$

其中：

- `N`：参数量。
- `D`：训练 token 数。

推理 FLOPs 与生成 token 数、模型参数量、上下文长度有关。decode 阶段每 token 需要过所有层，但 attention 还要读历史 KV。

## 激活显存

激活显存受这些因素影响：

- batch size。
- sequence length。
- hidden size。
- layer 数。
- 是否保存 attention matrix。
- 是否 activation checkpointing。

优化手段：

- gradient checkpointing / activation recomputation。
- FlashAttention。
- sequence parallel / context parallel。
- 减小 micro batch。

## 常见 OOM 位置

| 场景 | 可能原因 |
|---|---|
| SFT OOM | seq_len 太长、batch 太大、LoRA rank 高 |
| RL OOM | actor/ref/critic/reward 多模型并存 |
| 推理 OOM | KV Cache 长上下文或并发过高 |
| checkpoint OOM | 保存/加载时临时聚合权重 |
| eval OOM | 忘记关梯度或 batch 太大 |

## 面试高频问题

1. **为什么 LoRA 省显存？**  
   冻结大部分 base 权重，不为其保存梯度和优化器状态。

2. **为什么 RLHF 比 SFT 更耗资源？**  
   RLHF 需要 rollout、reference、reward、critic 等额外模型/前向。

3. **为什么长上下文推理贵？**  
   prefill attention 计算大，decode 还要维护和读取大量 KV Cache。

4. **ZeRO/FSDP 主要省什么？**  
   分片参数、梯度和优化器状态，降低单卡显存压力。
