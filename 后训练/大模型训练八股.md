# 大模型训练八股

## 1. 大模型训练阶段？

**答案：** 常见训练链路可以按“能力形成 -> 行为塑形 -> 偏好优化 -> 业务适配 -> 评估上线”理解。预训练用海量 token 做 next-token prediction，形成语言、知识、代码和推理底座；SFT 用高质量指令数据让模型学会助手格式和任务输出；偏好优化或 RLHF 让模型在多个可行回答里更偏向高奖励答案；业务微调或继续训练补足垂直场景；最后用通用评测、业务评测、安全评测和线上反馈闭环。  
> **面试表达：** 预训练解决“会不会”，SFT 解决“像不像助手”，RL/偏好优化解决“哪个答案更好”，业务训练解决“在我的场景里能不能稳定做对”。

## 2. RL 相比 SFT 的收益体现在哪里？

**答案：** SFT 本质是最大似然模仿数据中的标准答案，虽然模型会泛化，但训练目标主要还是把概率质量推向数据分布里的示范行为；RL 可以直接优化奖励，把最终正确性、人类偏好、工具执行成功率、代码测试通过率等目标纳入训练。收益最明显的场景是数学、代码、Agent 任务这类“结果可验证但高质量轨迹不唯一”的任务，RL 能在 verifier/reward 可靠时通过采样探索出比 SFT 示范更优的解法。  
> **面试表达：** SFT 是 imitation，RL 是 optimization；SFT 教模型怎么答，RL 让模型为了 reward 主动选择更优答案。

## 3. 后训练中如何融合 math、code、中文、agent 等多个领域或方向的数据？

**答案：** 先把数据统一成稳定的 chat template，再按任务类型设计 loss mask、采样权重和评估集。混合方式可以是一次性 mixture，也可以分阶段 curriculum：先通用指令和格式，再 math/code 等可验证任务，最后加入 agent/tool 数据。关键不是简单拼数据，而是控制每类数据的比例、长度、质量、模板一致性和评估指标，避免某类数据凭 token 数或 loss 尺度主导训练。  
> **面试表达：** 多域后训练要同时管三件事：模板统一、采样配比、分域评估。

## 4. 多领域数据混训时，如何避免某个能力提升导致其他能力下降？

**答案：** 常用做法是保留通用能力 replay 数据，设置分域验证集和 early stop，用采样权重控制强势领域，用较小学习率或 adapter 降低更新冲击，并持续观察通用能力、目标能力、格式遵循和安全指标。如果发现某个领域提升但通用能力下降，要判断是数据比例过高、模板冲突、过拟合、loss mask 错误，还是 reward 设计把模型推向了狭窄策略。  
> **面试表达：** 不能只看目标域分数，必须有“护栏评测”和 replay，否则很容易做成灾难性遗忘。  

**相关：** [[后训练/SFT/LoRA 和 QLoRA|LoRA 和 QLoRA]]、[[预训练/Causal LM Objective 和数据工程|Causal LM Objective 和数据工程]]

## 5. SFT 冷启动和后续 RL 的关系是什么？为什么很多 RL 任务前需要 SFT 冷启动？

**答案：** SFT 冷启动给 policy 一个可用的初始分布，让模型已经会遵循指令、输出合法格式、调用工具或写出基本推理链。没有冷启动时，RL rollout 可能大量无效，reward 稀疏且方差很大，训练会把算力浪费在学习最基础的格式上。SFT 后再做 RL，相当于先把搜索空间缩小到合理行为附近，再用奖励优化细节。  
> **面试表达：** SFT 先把模型带到“会做题”的区域，RL 再在这个区域里优化“做得更好”。

## 6. Base model 已经具备一定 Agentic 能力时，为什么还需要构造业务 Agentic 数据继续训练？

**答案：** Base model 的 Agentic 能力通常是通用的，它可能知道“工具调用”这个模式，但不一定理解业务 API、权限边界、错误恢复、返回字段、公司内部术语和真实 workflow。业务 Agentic 数据能把工具 schema、上下文组织、异常处理、停止条件和最终答案格式对齐到具体场景，让模型不只是会调用工具，而是能在业务约束下稳定完成任务。  
> **面试表达：** 通用 Agent 能力解决“会不会调用工具”，业务 Agentic 数据解决“能不能按业务规则正确调用”。

## 7. 业务基模和通用基模有什么区别？业务基模效果应该如何评估？

**答案：** 通用基模追求广泛语言、知识和推理能力；业务基模在通用能力上叠加业务知识、业务流程、工具协议和输出风格。评估时不能只看通用 benchmark，要同时看业务任务成功率、关键字段准确率、工具调用正确率、延迟成本、安全合规、长尾失败和线上 A/B。通用能力也要保留一组回归评测，防止业务训练损伤基础能力。  
> **面试表达：** 业务基模不是“通用模型加点数据”这么简单，评估要按真实业务闭环设计。

## 8. 如何判断一个后训练策略能否从调研模型迁移到内部自研模型？

**答案：** 先比较两边的模型规模、tokenizer、chat template、Dense/MoE 结构、上下文长度、预训练能力和当前训练阶段，再做小规模控制实验。迁移时重点看 reward 是否同义、采样温度和长度是否可比、KL 是否稳定、loss mask 是否一致、学习率和 batch/token 数是否需要按规模重调。如果小实验中 reward 上升但通用评测、KL、长度或格式失控，说明策略不能直接照搬。  
> **面试表达：** 能不能迁移不是看论文分数，而是看模型分布、数据协议、训练阶段和稳定性指标是否对齐。

## 9. 调研用模型和目标模型在参数量、Dense/MoE 结构、训练阶段不同时，策略迁移需要注意什么？

**答案：** 参数量不同会影响学习率、batch size、KL 系数和 reward scale；Dense 与 MoE 不同会影响路由、通信、expert 负载和 logprob 稳定性；训练阶段不同会影响模型是否已经具备指令、推理或工具能力。迁移时应优先复现数据处理、模板、loss mask 和评估，而不是机械复用超参；MoE 模型还要额外监控 router 分布、expert 负载和 token-level ratio。  
> **面试表达：** 迁移策略要迁移“思想”和“验证指标”，不要盲目迁移超参。

## 10. MoE 和 Dense 模型在 RL 训练中有什么差异？

**答案：** Dense 模型每个 token 经过同一套 FFN，logprob 和梯度路径更稳定；MoE 每个 token 只激活部分 expert，RL 更新会同时改变 token 分布和 router 分布。MoE 训练要处理 expert load balance、all-to-all 通信、路由抖动、部分 expert 过拟合，以及 token-level importance ratio 因路由变化变得更噪的问题。  
> **面试表达：** Dense RL 主要看策略更新稳定性，MoE RL 还要看“路由和专家负载是不是也稳定”。

## 11. MoE 做 RL 为什么可能更不稳定？

**答案：** RL 会强化少数高奖励轨迹，这可能让 token 分布快速偏移，进一步改变 router 选择，导致 expert 负载不均、部分 expert 被过度更新或长期得不到训练。MoE 还有 all-to-all 通信和路由随机性，token-level logprob ratio 在长 CoT 或多轮 rollout 中更容易抖动。如果 KL、负载均衡和学习率控制不好，可能出现 KL 飙升、expert collapse 或某些能力突然退化。  
> **面试表达：** MoE 的不稳定来自“双重分布漂移”：policy 输出在漂，router/expert 分配也在漂。
